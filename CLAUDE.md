## Harness Engineering 적용 원칙

### 핵심 정의
> "AI가 실수할 때마다, 같은 실수가 구조적으로 재발하지 못하도록 환경을 바꾸는 것"

### Claude(나)의 반복 실수와 구조적 방지책

| 반복 실수 | 방지책 |
|-----------|--------|
| 기획 안 끝났는데 코드 작성 | 단계 게이트 (아래 참고) |
| 결정사항 문서 미저장 | 세션 종료 전 PLANNING.md/PROGRESS.md 업데이트 필수 |
| 화면 논의 없이 UI 만들기 | 설계 게이트 통과 전 코드 금지 |
| 사용자 확인 없이 진행 | 각 게이트에서 명시적 승인 필요 |
| 여러 단계 동시 진행 | 한 번에 하나만 |

---

## 단계 게이트 (절대 규칙)

```
[기획 게이트]
조건: PLANNING.md의 모든 섹션 완성 + 사용자 승인
통과 전: 설계 작업 금지

[설계 게이트]  
조건: 화면별 설계 문서 완성 + 사용자 승인
통과 전: 코드 작성 금지

[개발 게이트]
조건: 해당 기능의 완료 기준 명시 + 사용자 확인
통과 전: 해당 기능 코드 작성 금지
```

**현재 단계**: 기획 (화면 목록 논의 중)

## 세션 루틴

### 시작할 때
```
1. CLAUDE.md 읽기 (이 파일)
2. PLANNING.md 읽기 (기획 결정사항)
3. PROGRESS.md 읽기 (현재 진행 상태)
4. 현재 단계 게이트 확인
5. 사용자에게 현재 상태 브리핑 후 지시 대기
```

### 진행 중
- 결정사항이 생기면 즉시 PLANNING.md에 저장
- 한 번에 하나의 작업만

### 종료할 때
```
1. PLANNING.md 업데이트 (새로운 결정사항)
2. PROGRESS.md 업데이트
3. GitHub 커밋 + 푸시
```

---

## 완료 기준 (Definition of Done)

기능 하나가 "완성"되려면:
- [ ] 브라우저/앱에서 직접 눈으로 동작 확인
- [ ] 에러 없음 확인
- [ ] PROGRESS.md에 완료 기록
- [ ] GitHub 커밋
